## P-20260723-001 过早进入前端开发可能造成大规模返工

| 字段 | 内容 |
| --- | --- |
| 状态 | 已解决 |
| 发现日期 | 2026-07-23 |
| 解决日期 | 2026-07-23 |
| 类型 | 产品流程 / 返工风险 |
| 相关待办 | B-20260723-001、B-20260723-002、B-20260723-003、B-20260723-004 |

### Situation

电子衣橱 MVP 已完成初步功能范围讨论，但 PRD、信息架构、页面原型和视觉风格尚未确定。此时准备直接开始微信小程序前端实现，用户担心成品风格不符合预期，导致完成后再进行大面积修改。

### Task

在不产生前端代码返工的前提下，建立清晰的产品设计与开发闸门，并确认当前是否已经产生需要回滚的项目文件。

### Action

#### 原因分析

- 已确认的是产品愿景、MVP 边界和部分技术方向，不等于页面设计已经确认。
- 首页信息优先级、导航结构、核心操作路径和视觉语言仍存在多种合理解释。
- 如果先实现完整页面，视觉修改可能进一步影响组件结构、状态管理和接口设计，返工不只发生在样式层。

#### 考虑过的方案

| 方案 | 优点 | 风险或代价 |
| --- | --- | --- |
| 继续直接开发完整前端 | 能快速看到可运行页面 | 未确认的结构和风格会固化进代码，返工风险最高 |
| 只先确定颜色和视觉风格 | 能降低纯视觉返工 | 无法解决页面结构和用户流程可能错误的问题 |
| 先完成 PRD、用户流程、低保真原型和视觉方向，再开发 | 能用低成本产物提前暴露问题 | 正式编码开始时间稍后，但整体返工成本最低 |

#### 最终行动

采用第三种方案，立即暂停产品代码实现，将后续顺序调整为：轻量 PRD → 信息架构与用户流程 → 低保真原型 → 多个视觉方向对比 → 高保真原型 → 技术设计 → 分阶段开发。

### Result

- 已在任何项目代码或脚手架文件创建前暂停实现，因此无需回滚或删除代码。
- 已明确前端开发的启动条件：核心流程和视觉方向得到用户确认。
- PRD、原型和视觉设计已拆分为独立待办，后续可以逐项验证。
- 尚未产生可量化的开发工时节省数据，因此不声称具体节省比例。

### 经验总结

“功能清单已经讨论完成”不等于“可以直接编码”。对于视觉和交互占比较高的产品，应先用 PRD 与原型降低决策成本，再把确认后的结构固化进代码。

### Interview Summary

在个人电子衣橱项目准备进入开发时，我识别出需求范围虽然基本明确，但页面结构和视觉方向尚未经过验证。如果直接编码，后续风格调整可能同时影响组件结构和交互逻辑。我暂停了实现，对方案进行权衡后建立了 PRD、用户流程、低保真原型、视觉确认和技术设计的分阶段闸门。最终在尚未产生代码改动时消除了返工风险，并把后续工作转换成了可逐项验收的待办。

## P-20260723-002 `.git` 路径存在但项目仍不是有效 Git 仓库

| 字段 | 内容 |
| --- | --- |
| 状态 | 已解决 |
| 发现日期 | 2026-07-23 |
| 解决日期 | 2026-07-23 |
| 类型 | 开发环境 / 版本控制 |
| 相关待办 | 无 |

### Situation

项目需要建立本地版本历史，但执行 Git 状态检查时提示“not a git repository”。与此同时，文件系统检查又显示项目根目录已经存在 `.git` 路径，出现了互相矛盾的表象。

### Task

在不删除项目文档、不覆盖未知 Git 历史的前提下，确认 `.git` 的实际状态，安全初始化仓库，并提交当前 PRD 与项目日志。

### Action

#### 原因分析

- 只检查路径是否存在会误判仓库状态，因此进一步读取了 `.git` 的类型和目录内容。
- 检查确认 `.git` 是一个空目录，不包含 Git 仓库所需的元数据，也不存在需要保留的提交历史。
- 首次在受限环境中执行 `git init` 时，Git 无法向 `.git` 写入模板文件，报错为 `Permission denied`。
- 添加 GitHub 远程后，首次交互式推送长时间没有返回输出。后续证据表明提交实际已经到达远程，但交互进程没有及时结束；具体未返回原因尚未确认。

#### 考虑过的方案

| 方案 | 优点 | 风险或代价 |
| --- | --- | --- |
| 删除 `.git` 后重新初始化 | 操作直观 | 删除前无法假设目录一定没有需要保留的数据，破坏性更高 |
| 把项目文件移动到新目录 | 可以绕开当前目录状态 | 会改变项目路径，且没有解决权限根因 |
| 先确认目录为空，再直接初始化并申请必要写入权限 | 保留项目路径，不删除文件，处理范围最小 | 需要一次受控权限审批 |

#### 最终行动

采用第三种方案。确认 `.git` 为空后，在当前目录执行 `git init -b main`；受限写入失败后，通过受控权限审批重新执行初始化、暂存和首次提交。连接 GitHub 时先核验远程为空且当前账号具有推送权限；交互式推送未返回后停止等待，确认本机凭据已存在，再以非交互方式重试，并通过 GitHub API 读取远程提交验证结果。

### Result

- 已建立有效的本地 Git 仓库，默认分支为 `main`。
- 已创建首次提交 `e86c2a0 docs: initialize Closet DNA project`。
- 首次提交包含 `AGENTS.md`、PRD、待解决日志和问题日志，共 4 个文件。
- 已连接远程仓库 `https://github.com/staroseee/closet-dna.git`，本地 `main` 跟踪 `origin/main`。
- GitHub 远程已读取到提交 `0ea729c9e2e5cd64a78f342afeabfc25694e28ed`，证明首次文档历史已上传。
- 提交后工作区无未提交文件。
- 未删除或移动任何项目文档。

### 经验总结

`.git` 路径存在不代表仓库有效。遇到状态矛盾时，应先检查路径类型和内容，再选择最小且可恢复的处理方式；在受限执行环境中，还要区分仓库结构问题、元数据写入权限和认证交互问题。网络写操作没有及时返回时，不能直接假设失败或重复执行，应先核验远程状态并使用可观察的非交互重试。

### Interview Summary

在为项目建立版本控制时，我遇到文件系统显示 `.git` 存在、但 Git 判断当前目录不是仓库的矛盾状态。我没有直接删除目录，而是先检查其类型和内容，确认它只是一个没有元数据的空目录。初始化过程中又遇到受限环境拒绝写入 Git 元数据，我区分了仓库结构、权限和认证交互问题，通过受控授权完成初始化；首次推送未及时返回时，我没有盲目重复操作，而是核验凭据、采用非交互重试，并从 GitHub 读取提交 SHA 验证远程状态。最终保留了全部文档，建立了本地与远程一致的版本历史。

## P-20260723-003 完整外套抠图会遮挡内搭，无法表达敞开穿着

| 字段 | 内容 |
| --- | --- |
| 状态 | 待验证 |
| 发现日期 | 2026-07-23 |
| 解决日期 | 尚未解决 |
| 类型 | 产品 / 图像处理 / 搭配预览 |
| 相关待办 | B-20260723-009 |

### Situation

用户在评审“固定槽位选品 → 搭配预览 → 画布微调”流程时指出：上传并抠图后的外套通常是一整件二维图片，但现实中许多外套以敞开状态穿着。如果把完整外套图片直接覆盖在内搭上，会遮住本应可见的上衣，也无法正确表达身体、外套前片和内搭之间的空间关系。

### Task

明确搭配板与真人换装分别需要表达什么效果，并为外套敞开、扣合和叠穿建立不会误导用户的预览方案与验收边界。

### Action

#### 原因分析

- 单张完整外套图片只提供二维可见像素，不包含被外套遮挡区域的结构信息。
- 简单调整图片层级只能决定谁遮住谁，无法自动形成敞开外套左右前片之间的可见窗口。
- 搭配板用于选品规划，真人换装用于模拟上身，两者对真实性和处理成本的要求不同，不能默认为同一种预览。

#### 考虑过的方案

| 方案 | 优点 | 风险或代价 |
| --- | --- | --- |
| 搭配板采用平铺编排，不把外套伪装成已穿在身上 | 即时、稳定，不制造虚假上身效果 | 用户需要通过真人换装才能查看真实叠穿模拟 |
| 把外套分割为左右前片并生成中间透明区域 | 可以在二维画布表达敞开效果 | 对图片角度和服装结构敏感，自动分割容易错误 |
| 所有外套叠穿都直接调用真人换装 | 表达目标最接近真实上身 | 异步、成本高，不能承担每次快速搭配预览 |

#### 最终行动

采用分层方案：即时结果定义为平铺搭配板，不使用完整外套图片制造伪上身叠加；搭配记录外套敞开或扣合状态，并允许用户补充状态参考图；真实叠穿效果统一交给异步真人换装。单张外套自动拆分左右前片保留为后续实验，不作为 MVP 稳定能力。

### Result

用户已确认产品方案，PRD 已同步更新。低保真原型、换装输入传递和实际生成质量尚未验证，因此问题保持“待验证”。

### 经验总结

## P-20260723-004 当前工作区占用导致本地项目目录无法直接改名

| 字段 | 内容 |
| --- | --- |
| 状态 | 待验证 |
| 发现日期 | 2026-07-23 |
| 解决日期 | 尚未解决 |
| 类型 | 技术 / 开发环境 |
| 相关待办 | B-20260723-010 |

### Situation

产品和 GitHub 仓库已统一使用 `OOTDone / ootdone`，但把本地目录从 `D:\Projects\closet-dna` 改为 `D:\Projects\ootdone` 时，Windows 返回“文件正被另一进程使用”。当前 Codex 任务正以旧目录作为工作区，直接改名会中断当前会话。

### Task

在不丢失提交、不破坏当前任务的前提下准备新目录，并验证新副本与远程仓库一致；切换工作区后再安全删除旧目录。

### Action

#### 原因分析

改名前已确认源目录存在、目标目录不存在且两个路径都位于 `D:\Projects`。`Move-Item` 仍因进程占用失败，说明阻塞来自活动工作区句柄，而不是路径冲突或 GitHub 权限。

#### 考虑过的方案

| 方案 | 优点 | 风险或代价 |
| --- | --- | --- |
| 在当前会话中反复强制改名 | 表面步骤最少 | 可能中断工作区或造成不可控状态，不采用 |
| 退出当前工作区后手动改名 | 保留原目录的全部本地状态 | 当前任务无法继续自动验证 |
| 从已同步的远程仓库创建新目录，验证后再删除旧目录 | 可恢复、可验证，不影响当前任务 | 切换工作区后还需清理旧目录 |

#### 最终行动

选择第三种方案：从 `https://github.com/staroseee/ootdone.git` 创建 `D:\Projects\ootdone`，比较新旧仓库 HEAD、远程地址和工作树状态。旧目录暂时保留，等待用户切换到新工作区后再删除。

### Result

迁移检查点中新旧仓库 HEAD 一致，新仓库 `origin` 为 `https://github.com/staroseee/ootdone.git`，两个工作树均为空。用户在界面切换到新目录后再次尝试删除，旧任务的后台进程仍持有目录句柄；旧目录中的普通项目文件已经清理，只剩 `.git` 和 `.agents` 两个受占用目录。新目录完整可用且保存了全部内容；需要结束当前任务并从新目录创建任务后清理残留，因此问题保持“待验证”。

### 经验总结

待旧目录删除并完成最终验证后填写。
